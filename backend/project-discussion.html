NestJS Authentication API

A backend REST API with user registration, login, JWT-based authentication, and MySQL persistence — built with NestJS and TypeORM.

NestJS 11 TypeScript TypeORM MySQL JWT bcrypt class-validator @nestjs/config

1 Project Overview

This project is a backend authentication API — the server-side foundation for any application that needs user accounts. It handles everything from accepting a registration request to issuing a secure token that proves a user is logged in.

The project is part of a larger full-stack application (cities project). The backend/ folder contains all server code. A frontend/ folder exists and will be connected once the backend is complete.

3 Key Concepts Covered

🧩

Modular Architecture

NestJS organises code into Modules. Each module owns its own controllers, services, and providers. The root AppModule imports feature modules like AuthModule.

🔀

Dependency Injection (DI)

NestJS automatically creates and injects class instances (e.g. AuthService into AuthController). You declare what you need in the constructor; the framework supplies it.

🗄️

ORM & Entities

TypeORM maps a TypeScript class (User) to a MySQL table (users). Decorators like @Column() define each field. No raw SQL needed.

✅

DTO Validation

Data Transfer Objects (DTOs) define what shape incoming data must have. The ValidationPipe rejects requests that don't match before they reach the service.

🔐

Password Hashing

Passwords are never stored as plain text. bcrypt.hashSync() turns a password into an irreversible hash. bcrypt.compareSync() checks it at login without ever decrypting it.

🪙

JWT (Stateless Auth)

After login, the server signs a token containing the user's ID. The client sends it on every request. The server verifies the signature — no session storage needed.

🛡️

Guards

A Guard intercepts a request before it reaches the controller and decides whether to allow it. The JwtGuard blocks any request without a valid token.

🌍

Environment Variables

Secrets like the JWT key live in a .env file that is never committed to git. ConfigService reads them safely at runtime.

🏷️

TypeScript Decorators

The @ syntax (e.g. @Injectable(), @Entity()) adds metadata to classes and properties. NestJS and TypeORM read this metadata at runtime to wire everything together.

5 File-by-File Walkthrough

tsconfig.json TypeScript Configuration

Two settings are mandatory for NestJS and TypeORM decorators to work. Without them every @Column(), @Injectable() etc. throws a ts(1240) error. The module must be commonjs, not nodenext.

"experimentalDecorators": true,  // enables @Decorator() syntax
"emitDecoratorMetadata": true,   // TypeORM reads column types at runtime
"module": "commonjs"              // NestJS requires CommonJS — NOT "nodenext"
main.ts Application Entry Point

Boots the NestJS app and registers the global ValidationPipe. whitelist: true strips unknown fields from request bodies. transform: true converts raw JSON into typed DTO instances so decorators can run.

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe({
    whitelist: true,   // strips fields not declared in the DTO
    transform: true,   // turns plain JSON into a typed class instance
  }));
  await app.listen(3000);
}
app.module.ts Root Module

Loads .env globally, connects TypeORM to MySQL, and imports AuthModule. synchronize: true auto-creates the users table if it does not exist — it does not drop existing data on restart.

ConfigModule.forRoot({ isGlobal: true }),   // .env available everywhere
TypeOrmModule.forRoot({
  type: 'mysql', host: 'localhost', port: 3306,
  database: 'auth_db',
  entities: [User],         // every entity must be listed here
  synchronize: true,         // creates table on first run; safe for dev
})
user.entity.ts Database Table Definition

A plain TypeScript class decorated to become a MySQL table. TypeORM reads the decorators and creates exactly this schema. The unique: true constraint is enforced at the database level, not just application level.

@Entity('users')
export class User {
  @PrimaryGeneratedColumn()     id!: number;    // AUTO_INCREMENT
  @Column({ unique: true })     email!: string; // DB-level UNIQUE
  @Column({ type: 'varchar' })  password!: string; // stores hash only
  @Column({ type: 'varchar' })  firstName!: string;
  @Column({ type: 'varchar' })  lastName!: string;
}
create-user.dto.ts & login.dto.ts Input Validation

DTOs are not database entities. They only define what a valid HTTP request body looks like. The ValidationPipe in main.ts reads their decorators and automatically returns 400 Bad Request if validation fails — before the service is ever called.

// create-user.dto.ts
export class CreateUserDto {
  @IsEmail()                     email!: string;
  @IsString() @MinLength(6)      password!: string;
  @IsString() @IsNotEmpty()      firstName!: string;
  @IsString() @IsNotEmpty()      lastName!: string;
}

// login.dto.ts — only the two fields needed for login
export class LoginDto {
  @IsEmail()                     email!: string;
  @IsString() @MinLength(6)      password!: string;
}
auth.module.ts Auth Feature Module

Registers the User repository for injection (forFeature) and sets up the JWT module by reading the secret from .env via ConfigService instead of hardcoding it.

JwtModule.registerAsync({
  inject: [ConfigService],
  useFactory: (config: ConfigService) => ({
    secret:      config.get('JWT_SECRET'),       // from .env
    signOptions: { expiresIn: config.get('JWT_EXPIRES_IN') }, // '7d'
  }),
})
auth.service.ts Business Logic

The core of the application. Contains all logic — no HTTP concerns here, just data operations. The controller stays thin and only delegates to this service.

// REGISTER
async register(dto: CreateUserDto) {
  const existing = await this.userRepository.findOne({ where: { email: dto.email } });
  if (existing) throw new ConflictException('Email already in use'); // → 409

  const user = this.userRepository.create({
    ...dto,
    password: bcrypt.hashSync(dto.password, 10), // plain text → hash
  });
  await this.userRepository.save(user);           // INSERT into MySQL
  return { id: user.id, email: user.email, ... }; // never return password
}

// LOGIN
async login(dto: LoginDto) {
  const user = await this.userRepository.findOne({ where: { email: dto.email } });
  if (!user || !bcrypt.compareSync(dto.password, user.password))
    throw new UnauthorizedException('Invalid credentials'); // → 401

  const token = this.jwtService.sign({ sub: user.id, email: user.email });
  return { token, user: { id: user.id, email: user.email, ... } };
}
jwt.guard.ts Route Protection

Any controller method decorated with @UseGuards(JwtGuard) will be blocked unless the request includes a valid Authorization: Bearer <token> header. The decoded payload is attached to req.user for the controller to use.

canActivate(context): boolean {
  const token = request.headers.authorization?.split(' ')[1]; // "Bearer <token>"
  if (!token) throw new UnauthorizedException('No token provided');

  try {
    request['user'] = this.jwtService.verify(token, { secret });
    return true;                  // allow request through
  } catch {
    throw new UnauthorizedException('Invalid or expired token');
  }
}

8 Security Practices Applied

PracticeHow it is implemented
No plain-text passwords bcrypt hashes the password before storing. Even if the database is leaked, passwords cannot be reversed.
Password never returned The service manually selects which fields to return. The password field is always omitted from responses.
Vague login errors Login always returns "Invalid credentials" — never "email not found" or "wrong password" separately, which would let attackers enumerate accounts.
DTO whitelist whitelist: true in ValidationPipe strips any extra fields the client sends, preventing property injection attacks.
Secrets in .env The JWT secret is never in source code. It lives in .env which is excluded from git via .gitignore.
JWT expiry Tokens expire after 7 days (JWT_EXPIRES_IN=7d). An attacker who obtains a token cannot use it forever.
DB-level uniqueness unique: true on the email column is enforced by MySQL itself, not only by application code.

9 What Could Be Next

1

Frontend — React + Vite

Scaffold the frontend/ folder with React. Build a Register page and Login page. Store the JWT in localStorage. Send it with every API request as Authorization: Bearer <token>. Enable CORS on the backend first (app.enableCors()).

2

Move DB Credentials to .env

MySQL host, username, and password are currently hardcoded in app.module.ts. Move them to .env and read via ConfigService — the same pattern used for the JWT secret.

3

TypeORM Migrations

Replace synchronize: true with migration files for production. Migrations give full control over schema changes without risking data loss on a live server.

4

Refresh Tokens

Short-lived access tokens (15 min) + a long-lived refresh token stored in the DB. When the access token expires, the client exchanges the refresh token for a new one without logging in again.

5

Swagger / OpenAPI Documentation

Install @nestjs/swagger to auto-generate interactive API docs at /api. Useful for testing endpoints without needing Postman.

6

Rate Limiting

Install @nestjs/throttler to limit login attempts per IP (e.g. 5 per minute). Prevents brute-force attacks on the login endpoint.

7

Role-Based Access Control (RBAC)

Add a role column to the User entity (admin / user). Create a RolesGuard that checks req.user.role against a @Roles('admin') decorator on routes.
